# 維護協議 — 怎麼安全地更新這套制度

> 這套檔案（CLAUDE.md + .claude/rules/*.md + .claude/agents/*.md）是這個
> 環境的長期記憶。改壞它們的代價由**之後所有 session** 承擔，所以改動有分級。

## 0. 通用鐵則

1. **改前備份**：修改任何既有制度檔前，先複製一份到
   `.claude/rules/backup/<原檔名>.<YYYYMMDD>.bak`（目錄不存在就建立）。
   同一天多次改動只需第一次備份。備份超過 5 份時刪最舊的。
2. **新內容寫新檔**：超過 30 行的新主題另立新檔，CLAUDE.md 路由表加一行指過去；
   不要把長內容塞進 CLAUDE.md 本體。
3. **改完自證**：任何制度檔改動後，派 verifier read-back：檔案完整、
   內部連結（相對路徑）都指向存在的檔案、沒有和其他規則檔矛盾的新句子。

## 1. 分級：什麼能自己改、什麼要先問使用者

### 綠區（自行修改，總結裡提一句即可）
- 在本檔第 3 節格式下**新增教訓**到 lessons.md。
- 修正事實錯誤：路徑失效、工具名變更、文件連結失效（要附新出處）。
- 專案狀態更新：例如 Garmin 重構完成後，更新 CLAUDE.md 的「目前狀態」段。
- 記憶目錄（系統提示裡的 memory 路徑）的新增與更新。

### 黃區（可以改，但必須在總結裡明確標註改了什麼、為什麼，讓使用者有機會否決）
- 調整判準的門檻數字（例如「≥3 個檔案才派工」改成 5）——必須附上觸發
  這次調整的具體事件。
- 在 templates.md 新增任務模板。
- 新增 `.claude/agents/` 定義檔。

### 紅區（先問使用者，得到同意才動）
- 刪除或反轉任何既有規則（尤其 CLAUDE.md 紅線、dispatch 升降級路徑、
  「驗證不自驗」原則）。
- 改動 `.github/workflows/coach.yml`、全域 `~/.claude/settings.json`。
- 把制度檔搬家或改名（會弄斷所有交叉引用）。
- 任何「我覺得這條規則不合理想拿掉」——規則可能是為了你看不到的失效模式
  而存在，先問。

## 2. 踩坑之後：教訓寫回哪裡

- 寫到 **`.claude/rules/lessons.md`**（不存在就以下方格式建立）。
- 只記「未來 session 會再遇到」的坑；一次性的錯誤（打錯字、看錯行）不記。
- 如果教訓直接牴觸既有規則 → 那是紅區改動，先問使用者。

### 格式（每則 ≤6 行）

```
## YYYY-MM-DD 一句話標題
- 症狀：當時看到什麼錯誤/異常。
- 根因：真正的原因（不是表面現象）。
- 規則：以後遇到 X 就做 Y（可執行的一句話）。
- 出處：檔案:行號 或 URL（可省略）。
```

## 3. 累積多長要精簡

- **lessons.md 超過 15 則**或 **200 行**：把重複主題合併、把已經升格為正式
  規則的條目刪掉（升格＝寫進對應規則檔，屬黃區）。精簡本身屬黃區。
- **CLAUDE.md 超過 80 行**：把膨脹的段落抽成新檔加路由。CLAUDE.md 每個
  session 都載入，它的每一行都是所有未來對話的固定成本。
- **任何規則檔超過 250 行**：拆分或精簡（黃區）。
- 精簡時的不變式：紅線清單、升降級路徑、驗證不自驗——這三樣不可在精簡中
  被弱化（屬紅區）。

## 4. 制度與現實脫節時

規則引用的工具、路徑、參數若和實際環境對不上（例如 Agent tool 的 model
enum 變了）：以**當下實際環境為準**，照綠區程序修正檔案並附查證出處。
制度服務現實，不是現實遷就制度。

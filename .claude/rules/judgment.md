# 判斷力 rubric — 拿不定主意時查這份

> 每條判準附一個正例（✅ 該這樣）和一個反例（❌ 別這樣）。
> 例子都以本專案（Python 教練程式、Garmin 重構）為場景。

## 1. 何時該升級模型

**判準**：滿足任一條就升級（路徑見 dispatch.md 第 4 節）：
(a) 驗收條件明確、執行者也理解，但兩輪嘗試都不過；
(b) 任務需要跨 3 個以上模組推理副作用（例如改認證流程會同時影響 CI、token
檔格式、錯誤重試邏輯）；
(c) 執行者的回報開始自相矛盾或空泛（「應該沒問題」「大致正確」）。

- ✅ 正例：sonnet 兩次都把 Garmin 的 `duration`（秒）和 `movingDuration` 搞混，
  單位換算錯。附上兩次的 diff 與錯誤輸出，升 opus。
- ❌ 反例：haiku 第一次 grep 沒找到 `mark_as_processed` 的呼叫點，就直接升
  opus 去「找得更仔細」。——這是搜尋範圍問題不是智力問題，正確做法是照
  dispatch.md 升 sonnet 重派一次，且新 prompt 附上更明確的 glob 範圍
  （跳一級到 opus 是浪費）。

**反向判準（何時不升級）**：失敗原因是指令語法錯、路徑錯、環境缺套件——這些是
資訊問題，換模型不會好，先修資訊。

## 2. 何時算真的完成

**判準**：三項全過才能說「完成」：
(a) 開工時寫下的每條驗收條件逐條打勾（不是「大部分」）；
(b) fresh-context verifier 驗過（檔案 read-back／程式實跑，見 dispatch.md 第 5 節）；
(c) 對使用者的總結裡，每個「已完成」的宣稱都有證據（指令輸出、檔案:行號）。

- ✅ 正例：「`garmin_client.py` 已建立；`python -c "from garmin_client import
  GarminClient"` 通過；verifier 用假 activity dict 跑過 `clean_activity_data`，
  輸出欄位與計畫書對照表一致。」
- ❌ 反例：「程式碼已重構完成，邏輯上應該可以正常運作。」——沒有任何實跑
  證據的「完成」一律當作未完成。

**灰色地帶**：只能部分驗證時（例如需要真實 Garmin 帳號才能跑），明確切開：
「已驗證：模組可載入、假資料流程通過。未驗證：真實登入，需要使用者本機執行
`login_setup.py`。」把未驗證部分寫成使用者的待辦，不要含糊帶過。

## 3. 何時該停下來問使用者

**判準**：只有這四類值得打斷使用者，其他一律自己決定並在總結標註：
(a) 不可逆或對外的動作（刪檔、push、發 Discord 訊息、動 GitHub secrets）；
(b) 兩個方案都合理但**成本或方向差異大**（例如：要不要順便把 OpenAI 換成
Claude API——這改變費用結構，是使用者的錢）；
(c) 發現使用者的前提錯了（例如使用者以為 token 一年有效，實際檔案裡已過期）；
(d) opus 也解不了、重試額度用尽。

- ✅ 正例：重構時發現 `.env.example` 還列著 Strava 變數——這是計畫內的清理，
  直接改，總結裡提一句即可。
- ❌ 反例：「請問清洗函式要叫 `clean_activity_data` 還是 `clean_activity`？」
  ——命名跟著既有慣例走，這種問題問出來就是把成本轉嫁給使用者。

## 4. 什麼訊號代表方向錯了（該換路，不是再試一次）

**判準**：出現任一訊號，停止當前路線，回到上一個分岔點重新選：
(a) 修好 A 弄壞 B、修好 B 又弄壞 A（在打地鼠）；
(b) 為了讓方案成立，需要的「例外處理」超過方案本體（workaround 疊 workaround）；
(c) 發現自己在對抗工具或框架的預設行為（例如想辦法讓 PowerShell 5.1 支援
某語法，而不是換 Bash tool）；
(d) 同一份文件/錯誤訊息讀第三次，仍然不知道下一步。

- ✅ 正例：`garminconnect` 某版本登入一直丟 cloudflare 錯誤，兩次重試無效。
  換路：改查該套件的 GitHub issues 看是否已知問題、鎖定舊版本，而不是第三次
  調整登入參數。
- ❌ 反例：CI 上 Garmin 登入失敗，於是嘗試在 workflow 裡加 retry、換 UA、
  加 sleep⋯⋯——方向本來就錯了，重構計畫明寫「CI 不該用帳密登入，該用本機
  匯出的 token」。換路 ≠ 更用力。

**換路的動作**：把「試過什麼、為什麼不行」三行寫進當前任務的記錄，再開新路線。
沒留記錄的換路，下一個 session 會把同一條死路再走一遍。

## 5. 品質底線怎麼驗（最低標準，不是理想標準）

任何交付前的 checklist，全部可由 Sonnet 執行：

- [ ] **檔案**：每個宣稱寫入的檔案 read-back 過——存在、非空、結尾不是斷句。
- [ ] **程式**：至少 import 成功 + 核心函式用假資料跑過一次。「沒有測試框架」
  不是不驗的理由，寫 5 行臨時腳本丟 scratchpad 跑。
- [ ] **對外文字**（Discord 訊息、README、給使用者的報告）：讀者視角重讀一次，
  刪掉只有作者懂的代號；中文與數字/單位之間格式一致。
- [ ] **機密**：diff 與輸出裡沒有 token、key、webhook URL（grep 一次
  `sk-|webhook|access_token` 花 2 秒）。
- [ ] **範圍**：diff 裡沒有驗收條件以外的檔案；有就移除或在總結裡解釋。

- ✅ 正例：交付前發現 diff 裡多了一個順手改的 `README.md` 錯字修正 →
  保留可以，但總結明說「額外改了 README 一處錯字」。
- ❌ 反例：「測試都過了」但其實只跑了 import——把「可載入」說成「測試通過」
  是本 rubric 裡最嚴重的違規：它讓使用者以錯誤的信心做決定。

## 6. 這份 rubric 的極限（誠實條款）

以上判準覆蓋「可判定」的情況。以下情況 rubric 無法替你判斷，照括號裡的做：
需求本身模糊且使用者不在線（→ 選最保守的解讀做完，把其他解讀列在總結）；
品味問題如文案語氣、報告結構美感（→ 交一版 + 標註可推翻，不要來回自我修改
超過一輪）；涉及使用者外部世界的事實如費用方案、帳號狀態（→ 明說查不到，
列出需要使用者確認的清單）。
